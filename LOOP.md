# 競馬予想 自律ループ運用書（LOOP.md）

このファイルは、人が介在せず「データ取得→予想生成→結果記録→指標更新→レポート」を回す自律ループの手順とガバナンス。cron / GitHub Actions / 手動、どの起動機構でも本書に従う。**判断の指示書は CLAUDE.md（特に §9〜§13）。本書は"回し方"のみ。**

## 0. 最重要＝ハード制約（自動執行・ループは自分で緩められない）
[[feedback_fitcockpit_safety_net_engineering]]の原則を移植。以下は破ってはならない不変条件：
1. **ルール・パラメータ・構造を自動変更しない**：§13・ゲート・消去ホワイトリスト・ブレンドw・softmaxβ・EV閾値・Kelly λ を**ループが書き換えることは禁止**。変更が要ると判断したら、根拠を `loop-report.md` に**「提案」として書くだけ**（適用は人間承認）。
2. **緩める方向のチューニング禁止**：的中率がよく見えるよう閾値を緩める、見送り基準を甘くする等は絶対にしない。
3. **結果を見た後に p_win を動かさない**：発走前に確定した p_win は、結果確定後に retro-fit しない（発走前の馬場の階級変化=§9-3 は例外的に発走前更新可）。
4. **データを捏造しない**：取得失敗時は空欄のまま。想定値でp_win/オッズ/結果を埋めない。
5. **取得失敗＝停止＋ping**：自動取得に失敗したら作業を止め、`loop-report.md` に失敗を記録し、ユーザーに「○○が取得できず。貼ってください」とpingする（graceful degradation）。
6. **評価ウィンドウの凍結を守る**：Winner-First(v3.0/2026-07-12開始)の20-30R期末まで、ルール改変は提案のみ。

## 1. 対象レースの特定
1. `natsu-juusho.html`（夏）/ `autumn-watch.html`（秋）の R 配列から、**今日から3日以内 or 今週末の重賞**を特定。
2. **★★★のみ本ループの深掘り対象**。★★=軽量、★=原則スキップ（薄データ2歳等）。
3. 複数あれば全て処理。日付は毎回WebSearchで再確認し、カレンダーとズレたら修正（捏造禁止）。

## 2. ステージ自動判定（何が出ているかで分岐）
対象レースについて、**取得できた情報で現在ステージを判定**し、そのステージの作業だけ行う（重複作業を避ける）：

| ステージ | トリガ（取得できたもの） | 作業 |
|---|---|---|
| **A. 事前** | 登録馬のみ（枠順まだ） | フォルダ雛形＋待ち受け（コース傾向・過去傾向・登録馬）。捏造せず「発表待ち」 |
| **B. 枠順後** | 枠順・確定斤量・騎手 | **全9視点を実データで深掘り**（①展開〜⑨騎手・血統含む）＋Winner-First関門判定。印は仮 |
| **C. オッズ後** | 単勝オッズ | Winner-First確定＋市場ブレンドEV＋CLV＋**メス（具体実績＞推測overlay）**＋二層買い目 |
| **D. 当日** | 馬場発表・馬体重・締切直前オッズ | 馬場急変を最優先反映（§9-3）＋CLV最終補正 |
| **E. 結果後** | 着順・払戻 | **track.htmlの較正記録にp_win/結果を記入**（第◯点）＋教訓ログ（あれば・提案として）＋index/prediction結果反映 |

## 3. 各ステージの手順（判断はCLAUDE.md準拠）
- **取得元**：JRA公式・netkeiba・競馬ラボ・ウマニティ等をWebFetch/WebSearch。日本語ファイル名は要percent-encoding。
- **B枠順後**：CLAUDE.md §12の重み（近走20/指数17で地力37%…）で全馬にp。§13-Bホワイトリスト厳守（ハード制約以外で有力馬を消さない＝格下げまで）。
- **C オッズ後**：市場pはオーバーラウンド補正（(1/O)/Σ）。§13勝ち馬先行＝的中は勝ち本命/妙味は相手。§13-Aアンカーは§10-3スポットのみ。**§13-Dエッジ源泉（展開/レース質）を1行明記**。§9-10・七夕賞教訓＝**具体的当舞台実績＞推測overlay**。
- **E 結果後**：`track.html`の`WF`配列の該当レースに `winner` を記入（cands外の勝ち馬なら `winnerOdds` も）。paper/real・retを記入。**self LL/mkt LL/ゲート較正が自動計算される**。期末まで判定は保留（区間で読む）。

## 4. デプロイ（毎回）
1. `sw.js` の `CACHE` 版を上げる＋新規ファイルは `ASSETS` に追記。
2. prediction.htmlのバッジ版・index.htmlのカード/バッジ版を更新。
3. `git add -A && commit && push`（main直・§7で承認済）。GitHub Pages反映を確認。

## 5. レポート（毎回・必須）
`loop-report.md` に1行追記：`YYYY-MM-DD HH:MM ｜ レース ｜ ステージ ｜ 実行内容 ｜ 取得成否 ｜ commit ｜ 提案(あれば)`。
- **変更提案**があれば同ファイルに「提案：〜（根拠：較正データ〜）」と書く＝人間が承認するまで適用しない。

## 6. 起動機構（3択・恒久性が違う）
- **A. CronCreate（セッション内）**：このClaudeセッションが起きている間だけ発火・**7日で失効**。短期デモ・今週末1回に向く。
- **B. Windows タスクスケジューラ → ヘッドレス `claude -p`**：PCが起動していれば恒久。ローカル依存。
- **C. GitHub Actions（スケジュール workflow）**：クラウドで**真の無人・数ヶ月**。要 `ANTHROPIC_API_KEY` secret＋実行スクリプト。**恒久ループの本命**。

## 7. 期末（20-30R）にだけ人がやること
較正データ（self LL・ゲート較正・回収率）を根拠に、`loop-report.md`の提案を検討し、**必要ならCLAUDE.md §13を人手で改訂**（区間で読む・大破の検出のみ・市場超えの証明には非充分＝§13-C読み方の規律）。ここだけは自動化しない。
